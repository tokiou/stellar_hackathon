1. Propuesta y Objetivos

1.1 Visión del Proyecto

Intent Wallet Copilot es un asistente de wallet inteligente para Solana que permite a los usuarios expresar intenciones de transacciones en lenguaje natural. El sistema parsea el texto, evalúa riesgos de seguridad mediante reglas determinísticas, presenta previsualización detallada y permite al usuario firmar con su propia wallet.

1.2 Objetivos Principales

  • Simplicidad UX: Permitir transacciones complejas mediante comandos simples en lenguaje natural
  • Seguridad first: Análisis de riesgo multinivel antes de cada transacción
  • No-custodial: Las claves privadas permanecen siempre en la wallet del usuario
  • Transparencia: Previsualización completa de cada transacción antes de firmar
  • MVP funcional: Transferencias reales + demo de swaps (preparación para Jupiter)

1.3 Acciones Soportadas

✅ Transferencias (Transfer)

Estado: Funcional

  • SOL nativo
  • Tokens SPL (USDC, BONK, JUP, PYTH)
  • Creación automática de ATAs
  • Verificación de balance pre-transacción

⚙️ Swaps (Swap)

Estado: Modo Demo

  • Quotes simulados con precios fijos
  • Flujo UX completo implementado
  • Preparado para integración Jupiter
  • Firmas simuladas (demo_signature)
ℹ️ Nota importante: El proyecto NO utiliza LLMs para parsing. Todo el parseo de intents es determinístico mediante expresiones regulares y reglas explícitas.

4. Flujo de Usuario (UX)

4.1 Journey Completo

  1. Landing sin wallet conectada:
    • Hero section con título "Intent Wallet Copilot"
    • Botón "Connect Wallet" prominente
    • Explicación de qué hace la app
  2. Usuario conecta wallet (Phantom en Devnet):
    • Header muestra estado conectado con dirección truncada
    • Badge verde "Connected" con indicador de Devnet
    • Aparece el input de intents y avisos de seguridad
  3. Usuario escribe un intent:
    • Ejemplos: "Swap 0.1 SOL to USDC" o "Send 5 USDC to <address>"
    • Submit → flowState cambia a 'parsing' (spinner 400ms)
  4. Sistema parsea y evalúa:
    • Parser regex extrae: action, tokens, amount, recipient/slippage
    • Si falla → muestra ParseError en panel rojo
    • Si OK → genera TransactionPreview (quote o transferPreview)
    • Risk engine evalúa: LOW/MEDIUM/HIGH/BLOCKED
  5. Usuario revisa preview:
    • ParsedIntentPanel: muestra intent parseado con badge de confianza
    • TransactionPreviewPanel: detalles del quote/transfer, fees, exchange rate
    • SafetyReviewPanel: nivel de riesgo con razones detalladas
  6. Confirmación y firma:
    • Si BLOCKED → botón deshabilitado, mensaje de bloqueo
    • Si HIGH → requiere escribir frase de confirmación
    • Si MEDIUM/LOW → botón directo "Prepare Transaction"
    • Al confirmar → flowState 'signing', se construye Transaction
  7. Ejecución:
    • Transfer: checkBalance → buildTransaction → sendTransaction → confirmTransaction
    • Swap (demo): buildSwapTransaction retorna null → simula firma → demo signature
    • Actualiza historial en localStorage
  8. Resultado:
    • Success → muestra signature con link a Solana Explorer
    • Error → muestra mensaje de error (rechazo, falta balance, etc.)
    • Botón "Start New Transaction" para resetear flujo

4.2 Panel de Historial

Accesible desde Header (toggle button). Muestra últimas 50 transacciones del usuario en este navegador:

  • ID único, timestamp, texto original
  • Tipo de acción (swap/transfer)
  • Nivel de riesgo evaluado
  • Estado: pending/confirmed/failed/cancelled
  • Signature (si existe) con link a Explorer

5. Máquina de Estados

5.1 AppFlowState

El flujo de la aplicación está gobernado por flowState en Index.tsx:

Estado Descripción Siguiente Estado
idle Estado inicial, input disponible → parsing al submit
parsing Ejecutando parseIntent (400ms delay) → parsed (OK) o error (fallo)
parsed Intent parseado, generando preview → reviewing
reviewing Mostrando preview y riesgos al usuario → signing al confirmar
signing Construyendo y firmando transacción → success (confirmada) o error
success Transacción confirmada en blockchain → idle al resetear
error Error en cualquier etapa → idle al resetear

5.2 TransactionStatus

Estado paralelo que trackea el ciclo de vida de la transacción en blockchain:

  • pending - No iniciada
  • signing - Esperando firma del usuario en wallet
  • submitted - Enviada a RPC, esperando confirmación
  • confirmed - Confirmada en blockchain
  • failed - Falló por cualquier razón
  • cancelled - Usuario rechazó en wallet

6. Módulos Clave

6.1 intentParser.ts

Responsabilidad: Parsear texto natural a estructuras tipadas sin LLM

Estrategia de Parsing

  • Regex patterns específicos para cada tipo de intent
  • Prioridad: primero intenta transfer (más específico), luego swap
  • Validaciones inline: tokens permitidos, formato de dirección, amounts positivos
  • Retorna ParseResult: { ok: true, intent } o { ok: false, error }

Patrones Soportados

// TRANSFERS
"Send 5 USDC to <address>"
"Transfer 0.01 SOL to <address>"

// SWAPS
"Swap 0.1 SOL to USDC"
"Buy 0.05 SOL of BONK"      // usar 0.05 SOL para comprar BONK
"Convert 10 USDC to JUP"
"Exchange 5 USDC for SOL"
"Buy JUP with 10 USDC"

Validaciones

Tipo Validación Error
Token Debe estar en allowlist (SOL/USDC/BONK/JUP/PYTH) unsupported_token
Amount Número positivo parseFloat válido invalid_amount
Address PublicKey.isOnCurve o formato base58 válido invalid_address
Same token Input ≠ Output en swaps parse_failure
Unsupported action Detecta keywords: stake, lend, nft, etc. unsupported_action

6.4 riskEngine.ts

Responsabilidad: Motor de análisis de riesgo basado en reglas

Niveles de Riesgo

  • LOW: Transacción estándar sin factores de riesgo elevados
  • MEDIUM: Tokens no-major, meme tokens, price impact > 3%
  • HIGH: Montos grandes (> 1 SOL equiv), price impact > 10%
  • BLOCKED: Parser confidence bajo, transacción bloqueada

Reglas de Evaluación (Swaps)

Condición Severidad Label
BONK involucrado MEDIUM "Meme token involved"
Token no-major (≠ SOL/USDC) MEDIUM "Non-major token"
Monto > 1 SOL equivalente HIGH "Large amount"
Price impact > 10% HIGH "Severe price impact"
Price impact 3-10% MEDIUM "Elevated price impact"
Parser confidence = low BLOCKED "Low parser confidence"

Confirmación de Riesgo Alto

Si nivel = HIGH, se requiere que el usuario escriba exactamente:

"I understand the risk and want to continue"

6.5 transactionBuilder.ts

Responsabilidad: Construir transacciones reales de Solana

buildTransferTransaction()

  1. SOL nativo: SystemProgram.transfer con lamports
  2. SPL tokens:
    • Deriva sender ATA y recipient ATA
    • Verifica si recipient ATA existe (getAccount)
    • Si no existe: agrega createAssociatedTokenAccountInstruction
    • Agrega createTransferInstruction
  3. Obtiene blockhash reciente y lastValidBlockHeight
  4. Asigna feePayer = sender
  5. Retorna Transaction lista para firmar

buildSwapTransaction()

// Actualmente retorna null
// Al integrar Jupiter:
// 1. Llamar Jupiter API /quote con params del intent
// 2. Obtener serialized transaction
// 3. Deserializar y retornar Transaction
return null;

checkBalance()

Verifica balance antes de transfer:

  • SOL: getBalance, reserva 0.01 SOL para fees
  • SPL: getAccount en ATA, convierte de raw amount a decimales
  • Retorna { sufficient: boolean, balance: number }

confirmTransaction()

Espera confirmación usando blockhash + lastValidBlockHeight strategy (preferido en v2)

6.6 history.ts

Responsabilidad: Persistencia de historial en localStorage

Funciones

  • loadHistory() - Lee array de HistoryEntry desde localStorage
  • saveHistoryEntry(entry) - Agrega nueva entrada al inicio, limita a 50
  • updateHistoryEntry(id, updates) - Actualiza status/signature de entrada existente
  • clearHistory() - Limpia todo el historial
  • generateId() - Genera ID único timestamp-based

Storage Key

9.2 Plan de Integración Jupiter

Paso 1: Obtener Quote

// Reemplazar getSwapQuote() en quoteProvider.ts
async function getJupiterQuote(intent: ParsedSwapIntent): Promise<SwapQuote> {
  const params = {
    inputMint: TOKEN_REGISTRY[intent.inputToken].mint,
    outputMint: TOKEN_REGISTRY[intent.outputToken].mint,
    amount: intent.amount * (10 ** TOKEN_REGISTRY[intent.inputToken].decimals),
    slippageBps: intent.slippage * 100,
  };

  const response = await fetch(
    `https://quote-api.jup.ag/v6/quote?${new URLSearchParams(params)}`
  );
  
  const data = await response.json();
  
  // Mapear data a SwapQuote interface
  return {
    inputToken: intent.inputToken,
    outputToken: intent.outputToken,
    inputAmount: intent.amount,
    estimatedOutput: data.outAmount / (10 ** TOKEN_REGISTRY[intent.outputToken].decimals),
    priceImpact: data.priceImpactPct,
    slippage: intent.slippage,
    route: data.routePlan?.map(r => r.swapInfo.label).join(' → ') || 'Direct',
    provider: 'Jupiter Aggregator',
    networkFeeEstimate: data.platformFee?.amount || 0,
    exchangeRate: /* calcular */,
  };
}

Paso 2: Construir Transacción

// Reemplazar buildSwapTransaction() en transactionBuilder.ts
async function buildSwapTransaction(
  connection: Connection,
  sender: PublicKey,
  intent: ParsedSwapIntent,
): Promise<Transaction> {
  // 1. Obtener quote de Jupiter
  const quoteResponse = await getJupiterQuoteRaw(intent);
  
  // 2. Llamar Jupiter swap endpoint
  const swapResponse = await fetch('https://quote-api.jup.ag/v6/swap', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      quoteResponse,
      userPublicKey: sender.toString(),
      wrapAndUnwrapSol: true,
    }),
  });
  
  const { swapTransaction } = await swapResponse.json();
  
  // 3. Deserializar transaction
  const transactionBuf = Buffer.from(swapTransaction, 'base64');
  const transaction = Transaction.from(transactionBuf);
  
  return transaction;
}

Paso 3: Ejecutar

El flujo en handlePrepare() quedaría:

if (parsedIntent.action === 'swap') {
  // Ya no simular: construir tx real
  const tx = await buildSwapTransaction(connection, publicKey, parsedIntent);
  
  // Firmar con wallet adapter
  const signature = await sendTransaction(tx, connection);
  
  setTxSignature(signature);
  setTxStatus('submitted');
  
  // Confirmar
  const confirmed = await confirmTransaction(connection, signature, ...);
  
  if (confirmed) {
    setTxStatus('confirmed');
    setFlowState('success');
  }
}

10. Persistencia (localStorage)

10.1 Esquema de Datos

Key: intent-wallet-copilot-history

Value: Array JSON de HistoryEntry

Estructura de HistoryEntry

interface HistoryEntry {
  id: string;                    // "1746825600000-ab7k3f9"
  timestamp: number;             // Date.now()
  originalText: string;          // "Swap 0.1 SOL to USDC"
  action: 'swap' | 'transfer';
  riskLevel: RiskLevel;          // 'LOW' | 'MEDIUM' | 'HIGH' | 'BLOCKED'
  status: TransactionStatus;     // 'confirmed' | 'failed' | etc.
  txSignature?: string;          // "3z8k..." o "demo_..."
  details: string;               // "0.1 SOL → USDC"
}

10.2 Límites

  • Max entries: 50 (las más antiguas se descartan)
  • Ordenamiento: Más reciente primero (unshift)
  • Tamaño aprox: ~5KB para 50 entradas (bien dentro de límite localStorage)

10.3 Ciclo de Vida de una Entrada

  1. Creación: Al parsear intent con éxito (status = 'pending' o 'failed' si BLOCKED)
  2. Actualización 1: Al obtener signature (updateHistoryStatus con sig)
  3. Actualización 2: Al confirmar o fallar (status = 'confirmed' | 'failed')
  4. No se eliminan: Quedan permanentes hasta clearHistory o límite de 50

10.4 Privacidad

⚠️ Consideración de privacidad: El historial incluye:
  • Direcciones de destinatarios completas
  • Montos de transacciones
  • Signatures públicas

Todo está en localStorage sin cifrar. Si múltiples usuarios comparten un navegador, se recomienda implementar clear history al desconectar wallet o añadir cifrado local.

12. Riesgos y Faltantes

12.1 Riesgos de Seguridad

Riesgo Severidad Mitigación Actual Acción Sugerida
Parser puede malinterpretar intents complejos MEDIUM Confidence score, preview obligatorio Agregar más test cases, considerar NLP library
Precios demo no reflejan mercado real MEDIUM Marcado como "Demo quote" Integrar oráculos (Pyth) o precios de Jupiter
Allowlist limitada (solo 5 tokens) LOW Bloqueo explícito de tokens no soportados Expandir con Jupiter strict list
Sin protección contra phishing en direcciones HIGH Validación base58, warning en safety review Agregar whitelist de direcciones conocidas
RPC público puede tener rate limits MEDIUM Ninguna Usar RPC privado en producción
Historial no cifrado en localStorage MEDIUM Disclaimer de uso personal Cifrar datos sensibles o usar sessionStorage

12.2 Funcionalidades Faltantes

  • ❌ Jupiter integration: Swaps reales on-chain
  • ❌ Multi-chain: Solo Solana, no Ethereum/BSC/etc.
  • ❌ Advanced intents: DCA, limit orders, conditional swaps
  • ❌ NFT support: Transfers de NFTs
