# Stellar Wave 6 Demo Runbook

This runbook proves the Compass Stellar thesis end-to-end on the public Testnet: a fresh account is created and funded, its native multisig is configured so Compass is a required signer, and all six demo cases run with their expected decision and observable on-network outcome. Every command below is **PLANNED** — the demo script does not exist yet (this is a planning wave). The runbook documents the intended operator experience so the script can be built against it.

> Reproducibility win: there is **no on-chain contract to deploy**. Stellar's native multisig (account signer weights + thresholds) enforces the policy gate at consensus level, unlike the Solana path which requires the Anchor program in `back/solana/agent-action-guard`.

## Prerequisites

- Node.js installed and `npm install` already run from the repo root.
- Public internet access to Stellar Testnet (Horizon, Soroban RPC, Friendbot). No private credentials or funds required — Friendbot funds the account.
- Waves 1–5 present (connectivity, decoder, operation mapping, cosigning/multisig, audit trail).
- A Compass cosigner key configured via the Wave 4 signer (used as the required co-signer; not generated by the demo).

## Step-by-step

### Step 1 — Create and fund a fresh Testnet account

```bash
# PLANNED — performed inside scripts/stellar-demo.mjs via the Wave 1 Friendbot helper
node scripts/stellar-demo.mjs --only=setup
```

Expected observable outcome: a new user/master keypair is generated and the account is funded by Friendbot. The script prints the account id and confirms a positive XLM balance on Horizon.

### Step 2 — Configure native multisig (Compass becomes required)

```bash
# PLANNED — submits a setOptions op: master weight + Compass key == medium threshold
node scripts/stellar-demo.mjs --only=configure-multisig
```

Expected observable outcome: a `setOptions` transaction sets signer weights and low/medium/high thresholds so that the master/user key and the Compass key **together** meet the medium threshold, and **neither alone** reaches it. The script then runs `inspectAccount` (Wave 4) and prints the signers, weights, and thresholds, asserting that Compass is genuinely required. If the assertion fails, the demo aborts.

### Step 3 — Case 1: legit payment within policy → ALLOW (executes)

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=1
```

Expected: brain returns **ALLOW**; Compass co-signs; combined weight meets the threshold; transaction **executes** on Testnet. An audit record is written (Wave 5).

### Step 4 — Case 2: payment to non-allowlisted destination → DENY

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=2
```

Expected: brain returns **DENY**; Compass does **not** sign; threshold unmet; transaction is **not executable**. Audit record written.

### Step 5 — Case 3: amount out of range → ESCALATE

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=3
```

Expected: brain returns **ESCALATE** (`REQUIRE_HUMAN_APPROVAL`); Compass does not auto-sign. Audit record written.

### Step 6 — Case 4: critical operation (setOptions / changeTrust) → ESCALATE

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=4
```

Expected: brain returns **ESCALATE** for the critical operation; Compass does not auto-sign. Audit record written.

### Step 7 — Case 5: user signs but Compass does NOT → threshold unmet (network rejects)

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=5
```

Expected: only the user/master signature is present; the account threshold is **not met**; on submission the network **rejects** the transaction — it is **not executable**. This is the structural proof of the thesis. Audit record written.

### Step 8 — Case 6: user + Compass sign → executable

```bash
# PLANNED
node scripts/stellar-demo.mjs --case=6
```

Expected: both the user/master signature and Compass's signature are present; combined weight meets the threshold; transaction is **executable** when submitted. Audit record written.

### Step 9 — Confirm Stellar tools are exposed through the existing MCP proxy

node scripts/stellar-demo.mjs --list-tools

Expected: the Stellar tools appear in `tools/list` with no modification to the MCP proxy core.

## Decisions (resolved open questions)

- **Run through the MCP proxy or call gateways directly?** DECIDED: default to running **through the existing MCP proxy** for fidelity (exercises the real routed surface); a `--gateway-direct` flag is offered as a faster fallback for iteration. Proxy core is never modified.
- **How is the "user" signature represented in an automated demo?** DECIDED: use a **second Testnet keypair acting as the user/master signer**, generated and funded by the demo in Step 1.
