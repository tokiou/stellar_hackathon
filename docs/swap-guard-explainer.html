🛡️ Arquitectura visible · Wallet Copilot

Guardrails para operar en Solana sin saltearse la capa de seguridad.

Esta app es una Next.js fullstack con separación física front/ y back/. El usuario conversa con un agente, conecta Phantom, aprueba propuestas y firma transacciones preparadas por backend. Antes de cualquier acción crítica, el sistema evalúa riesgo, políticas, precio, destino y condiciones on-chain/off-chain.

1. Mapa mental de la arquitectura

Idea central: una sola app Next deployada en Vercel, pero con carpetas separadas para que el código sea entendible: UI en front/, lógica server-side en back/, entrypoints oficiales en app/.
front/

UI React, hooks, stores, estilos, tipos de frontend y cliente API.

No guarda secrets. No llama providers externos.

app/

Next App Router: página principal y route handlers.

app/page.tsx importa front/src/App.tsx.

back/

Servicios server-side, tools del agente, RPC, providers y lógica de guardrails.

Usado por app/api/*.

infra/keeper

Supabase Postgres para cola auditable y Upstash QStash para invocar workers serverless.

Usado por órdenes condicionales recurrentes.

docs/

Referencia transversal, specs por feature y documentación de decisiones.

Cada feature vive en docs/<feature>/.

Browser / Phantom
  │
  ▼
front/src/App.tsx + components/hooks/stores
  │ consume /api/* mediante front/src/lib/api/client.ts
  ▼
app/api/*/route.ts  ── borde Next/Vercel
  │ valida input y delega
  ▼
back/services/*     ── providers, agente, RPC, guardrails
  │
  ├─ Solana RPC / Pyth / Orca / Helius / Birdeye / Jupiter
  ├─ Supabase Postgres: estado, locks, historial e idempotencia
  ├─ Upstash QStash: schedules firmados, retries y ticks serverless
  └─ Programas devnet: AgentActionGuard + ConditionalEscrowBuy

2. Qué corre dónde

ÁreaPuede hacerNo debe hacer
Frontend
front/src		 Renderizar UI, conectar Phantom injected, mostrar propuestas, firmar/enviar unsigned tx preparadas por backend. Consultar providers externos, calcular riesgo, construir transacciones desde intención, manejar API keys.
Route handlers
app/api/*		 Validar input HTTP, normalizar errores, llamar servicios backend. Duplicar lógica de negocio compleja o esconder contratos no documentados.
Backend services
back/services		 Evaluar guardrails, consultar providers/RPC, preparar unsigned tx, orquestar agente y tools. Firmar por el usuario o exponer secrets al browser.
Smart contracts
back/solana		 Enforcement verificable: approvals, políticas, condiciones y checks on-chain. Reemplazar todos los checks off-chain; se usan cuando hace falta garantía no bypassable.
Keeper infra
Supabase + QStash		 Persistir órdenes condicionales, tomar locks, auditar attempts e invocar el worker periódico. Ser fuente de verdad financiera: el contrato on-chain valida antes de ejecutar.

3. Flujo de guardrails antes de una acción crítica

1
Intención del usuario

El usuario pide transferir, swappear o crear una orden condicional desde el chat.

/api/chat
2
Interpretación y contexto

El backend/agent parsea intención, wallet actual, historial, parámetros y policy de usuario.

backend
3
Validación de riesgo

Se evalúa destino, token/protocolo, límites, frecuencia, slippage, precio, red y proveedores.

rules
4
Decisión

Resultado estable: permitir, bloquear o pedir confirmación reforzada con motivo claro.

ALLOW/WARN/REJECT
5
Preparación de transacción

Si corresponde, backend prepara una unsigned transaction y la UI la muestra como proposal.

unsigned
6
Firma con Phantom

El usuario firma/envía con Phantom injected. El backend puede recibir el resultado como proof/callback.

user signs
Regla de oro: ninguna operación crítica debe ejecutarse sin pasar primero por la capa de guardrails. Si falla, la app debe explicar el motivo y sugerir bloquear, confirmar extra o reintentar bajo nuevas condiciones.

4. Features principales

Transferencia a otra wallet

Valida formato, destino, reputación, allow/deny lists, señales externas y expiración/hash de la acción antes de preparar la tx.

Docs: wallet-safety-validation/ y agent-action-guard-guarded-transfer/

Swap a otro token

Evalúa quote, token, slippage, desviación de precio y puede usar guardrails on-chain con oracle antes del swap.

Docs: transaction-logic/, phantom-external-transaction-contracts.md

Orden condicional recurrente

Ejemplo: “comprame SOL cuando baje de 130 USD”. Usa escrow on-chain, oráculo Pyth y un keeper persistente con Supabase + QStash.

Docs: devnet-conditional-escrow-buy-sol/ y conditional-order-db-keeper/

Keeper condicional — ejecución sin procesos residentes

Usuario crea orden con Phantom
  └─ Orden/escrow queda on-chain
      └─ Backend registra conditional_orders en Supabase
          └─ QStash invoca /api/cron/conditional-orders cada 30s
              ├─ Worker toma batch con lock + TTL
              ├─ Refresca estado on-chain y precio Pyth
              ├─ Si no cumple: agenda next_check_at
              └─ Si cumple: intenta execute_order

El contrato valida nuevamente condición, staleness, expiración y estado.
La DB coordina; Solana decide.
Decisión de scheduling: usar dos schedules QStash por minuto: uno inmediato y otro con Upstash-Delay: 30s. Así se logra cadencia de 30 segundos sin setInterval ni worker residente en Vercel.

Swap Guard Oracle — caso de protección de precio

Usuario aprueba swap
  └─ Backend prepara TX con guardrails
      ├─ registra/valida action approval
      ├─ lee oracle Pyth SOL/USD
      ├─ compara quoted price vs oracle price
      └─ solo si pasa: ejecuta swap

Si la desviación excede el umbral:
  └─ la transacción falla/revierte y los fondos no se mueven.
Fuente de verdad de direcciones: docs/onchain-deployments.md. No hardcodear nuevas direcciones en UI sin documentarlas y revisar servicios backend.

9. Checklist de arquitectura sana

Antes de implementar
  • ¿La acción crítica pasa por guardrails?
  • ¿El frontend solo consume /api/*?
  • ¿Los secrets quedan en backend/Vercel?
  • ¿QStash/Supabase se usan solo server-side y con firma/locks?
  • ¿Hay contrato Zod/tipos si lo consume la UI?
Antes de review
  • npm test para frontend.
  • npm run test:back para backend/API.
  • npm run lint sin errores.
  • npm run build si tocaste rutas/config/imports globales.
