Mapa mental

Capas y responsabilidades

La regla de oro del repo es simple: el browser nunca maneja secrets ni decide riesgo crítico; pide propuestas al backend, firma con Phantom y la validación sensible vive server-side u on-chain.

🔐
Guardrail principal

Ninguna operación crítica debería ejecutarse sin pasar primero por validaciones de riesgo, política de usuario, contexto de red y, cuando aplique, enforcement on-chain.

Rutas API internas

Las rutas de app/api/* son el puente público interno. Desde ahí se importan servicios en BACK/services/*, evitando CORS y protegiendo API keys.

Criterios

Decisiones arquitectónicas vigentes

Una sola app Next en Vercel

El repo separa código, no deployments: FRONT/ y BACK/ conviven bajo Next App Router.

Backend como capa de guardrails

Providers externos, riesgo, construcción canónica de transacciones y policies viven detrás de /api/*.

Phantom firma, no decide

El frontend recibe propuestas o transacciones preparadas, y el usuario firma/envía desde la wallet.

On-chain cuando hace falta enforcement

Programas Anchor como agent-action-guard y conditional-escrow-buy cubren restricciones no bypassables.

DB + QStash para órdenes condicionales

La arquitectura propuesta usa Supabase como cola auditable y QStash como scheduler. No depende de setInterval ni de memoria efímera en Vercel.
